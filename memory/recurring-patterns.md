# 反复出现问题追踪表（自我学习引擎）

> 本文件是「举一反三」机制的核心数据源。
> 每次修复 bug 或解决问题时，检查此表是否有同类记录，有则升级计数并触发对应层级。

## 升级链触发规则

| 出现次数 | 触发动作 | 归属阶段 |
|----------|----------|----------|
| 1次 | 修复即审计：同文件/同模块搜索同类问题 | 反思 |
| ≥2次 | 全局审计：Grep 全项目扫描 + 建立防御规则 | 进化 |
| ≥3次 | 深度反省：根因链分析 + 规则加固 + 提议创建自动化检测(hook/lint/test) | 进化 |
| ≥4次 | 举一反三：类比推断 + 跨项目迁移 + 预防性建议 | 进化 |

> **操作规范**: 更新计数后，立即检查是否跨越触发阈值。跨越则在当前任务的进化阶段执行对应动作。

## 知识衰减策略（多维评分）

> 替代固定 90 天窗口，用多维评分决定模式状态转换。

```
衰减分 = 基础权重 × 访问频率系数 × 时间衰减系数 × 关联度系数

时间衰减系数 = e^(-0.01 × 天数)    // 约 69 天半衰期
访问频率系数 = log(1 + 访问次数)    // 越常触发越重要
关联度系数 = 活跃关联数 / 最大关联数  // 与其他活跃模式的连接

状态转换:
  活跃 → 休眠:   衰减分 < 0.3 且 30天未触发
  休眠 → 已根治: 衰减分 < 0.1 且 60天未触发
  已根治 → 归档: 衰减分 < 0.05 且无关联活跃模式
  复活: 任何状态均可因新匹配直接回到活跃
```

> **操作方式**: 每月底或每完成 10 个任务后，对所有模式执行一次衰减评分，更新状态。

## 进化效果度量

> 这些指标用于量化验证自我进化机制是否有效运作。理想状态：问题递减、首次解决率上升、升级深度下降。

| 指标 | 计算方式 | 当前值 | 理想趋势 |
|------|---------|--------|---------|
| 问题递减率 | count(pattern, 本月) / count(pattern, 上月) | - | < 1.0 |
| 规则命中率 | 自动规则拦截次数 / 总触发次数 | - | > 80% |
| 首次解决率 | 不经升级链解决 / 总问题数 | - | 持续上升 |
| 平均升级深度 | 问题平均在第几层被解决 | - | 持续下降 |

> 更新时机：每月底或每完成 10 个任务后更新一次。

**路由规则：**
- 成功率 < 70% → 升级 model（如 sonnet → opus）
- 成功率 < 50% → 触发策略审查（分析失败根因）
- 连续 3 次失败 → 自动降级兜底路径（规则兜底替代 LLM）
